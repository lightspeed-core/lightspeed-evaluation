# Output and Reporting

The framework produces structured evaluation reports in multiple formats with statistical analysis, visualizations, and optional database persistence.

## Behavioral Rules

### Report Formats

1. CSV reports provide a flat table with one row per metric evaluation result.
2. JSON reports provide the same data in structured format.
3. TXT reports provide a human-readable summary with pass rates and statistics.
4. All three formats are generated by default on every evaluation run.

### Statistical Analysis

5. EvaluationSummary computes statistics at four levels: overall, by-metric, by-conversation, and by-tag.
6. Each level includes: pass/fail/error counts, pass rate, mean score, and total tokens.
7. Confidence intervals (bootstrap method) are computed when sample size > 1.
8. Score distribution analysis identifies patterns across the result set.

### Visualizations

9. Graphs are generated per-configuration from the `enabled_graphs` setting.
10. Available visualizations include: pass rate charts, score distributions, heatmaps.
11. Visualization uses matplotlib and seaborn.

### Quality Score

12. A composite quality score can be computed from a configurable subset of metrics.
13. Quality score supports per-metric weights for weighted aggregation.
14. Quality score metrics must exist in the system config's metrics_metadata.

### Storage Backends

15. FileStorageBackend accumulates results in memory during evaluation and defers all disk writes (CSV/JSON/TXT/graphs) to `finalize()`. It requires the full evaluation dataset via `set_evaluation_context()` for cross-conversation report generation.
16. SQLStorageBackend commits results to a database (SQLite, PostgreSQL, or MySQL) immediately per conversation in `save_run()`. Its `finalize()` is a no-op (just logs a count).
17. CompositeStorageBackend chains multiple backends — a run can write to both file and database simultaneously.
18. Storage backends follow a lifecycle: initialize → save_run (per conversation) → set_evaluation_context → finalize → close.

### Token Usage Reporting

19. Token usage is tracked separately for API calls (agent inference) and judge LLM evaluations.
20. Per-judge token tracking is available in panel mode.
21. Token summary is printed to stdout at the end of evaluation.

## Configuration Surface

| Field/Flag | Type | Default | Description |
|---|---|---|---|
| `storage` | list | [file] | List of storage backend configs |
| `storage[].type` | string | — | Backend type: file, sqlite, postgresql, mysql |
| `storage[].connection_string` | string | — | Database connection URL (for sql backends) |
| `visualization.enabled_graphs` | list | — | Which graph types to generate |
| `quality_score.metrics` | list | — | Metrics included in quality score |
| `quality_score.default` | bool | — | Whether quality scoring is enabled by default |

## Constraints

- File backend requires SystemConfig access and the full evaluation dataset (`set_evaluation_context()`) for report generation.
- Database backend uses SQLAlchemy with URL encoding for special characters in connection strings.
- `finalize()` is called once after all conversations in the entire evaluation run complete. This matters for file storage (which defers all writes) but not for SQL storage (which persists incrementally).
