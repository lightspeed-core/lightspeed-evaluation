# Output and Reporting

The framework produces structured evaluation reports in multiple formats with statistical analysis, visualizations, and optional persistence to databases and Langfuse.

## Behavioral Rules

### Report Formats

- CSV reports provide a flat table with one row per metric evaluation result.
- JSON reports provide the same data in structured format.
- TXT reports provide a human-readable summary with pass rates and statistics.
- A quality report JSON (`*_quality_report.json`) is generated when quality scoring is configured, containing quality score, per-metric breakdowns with weights, and agent latency stats.
- All standard formats (CSV, JSON, TXT) are generated by default on every evaluation run.

### Statistical Analysis

- EvaluationSummary computes statistics at four levels: overall, by-metric, by-conversation, and by-tag.
- Each level includes: pass/fail/error counts, pass rate, mean score, and total tokens.
- Confidence intervals (bootstrap method) are computed when sample size > 1.

### Visualizations

- Graphs are generated per-configuration from the `enabled_graphs` setting.
- Available graph types: `pass_rates`, `score_distribution` (box plot with quartile spans), `status_breakdown` (pie chart), `conversation_heatmap`.
- Visualization uses matplotlib and seaborn.

### Quality Score

- A composite quality score can be computed from a configurable subset of metrics.
- Quality score supports per-metric weights for weighted aggregation.
- Quality score metrics must exist in the system config's metrics_metadata.

### Storage Backends

- FileStorageBackend accumulates results in memory during evaluation and defers all disk writes (CSV/JSON/TXT/graphs) to `finalize()`. It requires the full evaluation dataset via `set_evaluation_context()` for cross-conversation report generation.
- SQLStorageBackend commits results to a database (SQLite, PostgreSQL, or MySQL) immediately per conversation in `save_run()`. Its `finalize()` is a no-op (just logs a count).
- LangfuseStorageBackend accumulates results and writes scores to Langfuse on `finalize()`. Config supports inline `host`/`public_key`/`secret_key` or falls back to `LANGFUSE_HOST`/`LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` environment variables.
- CompositeStorageBackend chains multiple backends — a run can write to file, database, and Langfuse simultaneously.
- Storage backends follow a lifecycle: initialize → save_run (per conversation) → set_evaluation_context → finalize → close.

### Token Usage Reporting

- Token usage is tracked separately for API calls (agent inference) and judge LLM evaluations.
- Per-judge token tracking is available in panel mode via JudgeScore objects.
- Token summary is printed to stdout at the end of evaluation.

## Configuration Surface

| Field/Flag | Type | Default | Description |
|---|---|---|---|
| `storage` | list | [file] | List of storage backend configs |
| `storage[].type` | string | — | Backend type: file, sqlite, postgres, mysql, langfuse |
| `storage[].database` | string | — | Database name (for sql backends) |
| `storage[].host` | string | — | Database host for remote SQL backends (postgres, mysql) or Langfuse host URL; SQLite uses `database` for file path |
| `visualization.enabled_graphs` | list | — | Which graph types to generate |
| `quality_score.metrics` | list | — | Metrics included in quality score |
| `quality_score.default` | bool | — | Whether quality scoring is enabled by default |

## Constraints

- File backend requires SystemConfig access and the full evaluation dataset (`set_evaluation_context()`) for report generation.
- Database backend uses SQLAlchemy with URL encoding for special characters in connection strings.
- Langfuse backend requires the `langfuse` optional dependency.
- `finalize()` is called once after all conversations in the entire evaluation run complete. This matters for file storage (which defers all writes) and Langfuse (which writes scores), but not for SQL storage (which persists incrementally).
